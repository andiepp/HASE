# Runtime Architecture

## Overview

The HASE Runtime is responsible for connecting client applications, transports and hardware instruments through a common object model.

Its primary responsibilities are:

- hosting endpoint runtimes;
- maintaining runtime state;
- routing protocol requests;
- managing property values;
- executing commands;
- publishing events;
- remaining completely independent of the underlying transport.

The runtime never communicates directly over a serial port, TCP connection or BLE link. All communication is performed through transport implementations.

# Runtime Responsibilities

## Endpoint Management

The runtime owns one or more runtime endpoints.

Each endpoint contains:

- descriptor information;
- runtime instrument instances;
- connection state;
- transport association.

---

## Property Management

The runtime maintains the current cached value of every property.

The authoritative value always remains inside the instrument.

The runtime cache exists to:

- reduce protocol traffic;
- notify clients;
- detect changes;
- provide immediate access to current values.

Property updates may originate from:

- polling;
- notifications;
- commands;
- local simulation.

---

## Command Execution

Commands are routed through the runtime to the appropriate instrument.

The runtime is responsible for:

- parameter validation;
- routing;
- execution;
- returning protocol results.

---

## Event Distribution

Events generated by instruments are published through the runtime.

Subscribers may include:

- applications;
- automation services;
- logging;
- tracing;
- simulations.

---

# Protocol Integration

Protocol Version 1 is responsible only for message transport.

The runtime performs all business logic.

```
Protocol Message
        │
        ▼
BinaryProtocolPayloadCodec
        │
        ▼
Runtime Dispatcher
        │
        ▼
Runtime Endpoint
        │
        ▼
Runtime Instrument
```

The protocol layer never accesses instruments directly.

---

# Transport Independence

The runtime depends only on transport abstractions.

Consequently, the same runtime can operate over:

- loopback;
- serial;
- TCP/IP;
- BLE;
- MQTT;
- future transports.

Changing transport implementations never affects runtime behaviour.

---

# Variant-Based Runtime Values

Runtime values are transported using the protocol VariantSerializer.

Supported Protocol Version 1 types:

- null
- Boolean
- Int32
- Int64
- Double
- String

The protocol intentionally avoids CLR-specific binary serialization to remain platform independent and suitable for embedded systems.

---

# Property Values

Property values consist of:

- Variant value;
- UTC timestamp;
- Property quality.

The timestamp is encoded as Unix time in milliseconds.

Quality values are:

- Good
- Uncertain
- Bad

---

# Design Principles

The runtime follows several architectural principles.

## Separation of Concerns

Runtime, protocol and transport remain completely independent.

## Composition

Serializers build upon lower-level serializers rather than duplicating functionality.

## Platform Independence

The protocol does not depend on .NET-specific serialization.

## Testability

Every protocol component is independently unit tested.

## Extensibility

Additional transports, protocol versions and runtime services can be added without changing existing layers.
